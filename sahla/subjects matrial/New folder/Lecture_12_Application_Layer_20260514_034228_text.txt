Non-Persistent HTTP
e Each object (HTML file, image, etc.) is transferred over a separate TCP connection.
e After sending one object, the TCP connection is closed.
RTT (Round-Trip Time)
© RTT is the time it takes for a small packet to go:
e from client — server, and
e back from server — client.
e It includes propagation delay, transmission delay, and processing delay.
Response Time Components
1) One RTT to Initiate TCP Connection
e Client and server perform the TCP three-way handshake.
e This costs 1 RTT before any HTTP data is sent.
2) One RTT for HTTP Request and Initial Response
e Client sends the HTTP request (e.g., GET).
e Server sends back the first few bytes of the HTTP response.
© This costs another 1 RTT.
3) File Transmission Time
e The server transmits the entire requested file to the client.
e Time depends on:
e File size
e Link bandwidth
e This is not part of RTT, but added separately.

1. What Cookies Can Bring
HTTP is a stateless protocol, meaning the server does not remember previous requests from the same client.
Cookies are used to overcome this limitation by enabling websites to maintain state across multiple HTTP
transactions.
Cookies enable the following functionalities:
a) Authorization
e Cookies store authentication information (e.g., session IDs).
e After login, the browser sends the cookie with every request.
e The server uses it to identify the user without repeated logins.
b) Shopping Carts
e Cookies track items added to a shopping cart.
e The cart persists even when navigating between pages.
e Without cookies, the server would not know which items belong to which user.
c) Recommendations
e Cookies record browsing or purchase history.
© = This information is used to suggest products or content tailored to the user.
d) User Session State (e.g., Web E-mail)
e Cookies allow services such as webmail to remember.
e Logged-in user
e Open inbox
e Session preferences
e The session continues across multiple page requests.

[Image 2]  621×755px  (PNG)
OCR:
2. How to Keep “State” in the Web
Because HTTP itself is stateless, state must be maintained explicitly using additional mechanisms.
a) Protocol Endpoints Maintain State
e The sender (client) and receiver (server) maintain state information across multiple transactions.
e Example state information:
e User identity
e Session ID
e User preferences
b) Cookies Carry State in HTTP Messages
e Cookies are small pieces of data stored in the browser.
e Each HTTP request includes the relevant cookie in its header.
e The server reads the cookie and restores the session state.
In this way, HTTP messages themselves carry state, even though the protocol is stateless.
3. Privacy Implications of Cookies
a) Tracking and Profiling
e Cookies allow websites to track:
e Pages visited
e Time spent
e User behavior over time
e This enables detailed user profiling.
b) Personal Information
e Users may voluntarily provide:
¢ Name
e Email address
e When linked with cookies, this information allows sites to identify users personally, not just
anonymously.

1. What Is a Web Cache (Proxy Server)?
A Web cache, also called a proxy server, is an intermediary placed between clients (browsers) and origin
servers.
Its main goal is to satisfy client requests without contacting the origin server, whenever possible.
In the figure:
e The clients (browsers) are on the left.
e The web cache (proxy) is in the middle.
e The origin web servers are on the right.
© Red arrows show HTTP requests and responses passing through the cache.
2. How Web Caching Works (Step-by-Step)
Step 1: Browser Configuration
e The user configures the browser so that all HTTP requests are sent to the cache instead of directly to
the origin server.
Step 2: Client Sends Request to Cache
e The browser sends an HTTP request to the web cache (as shown by arrows from clients to the middle
box).
Step 3: Cache Lookup
The cache checks whether the requested object (HTML page, image, video, etc.) is already stored.
Case 1: Object Found in Cache (Cache Hit)
e The cache returns the object directly to the client.
e The origin server is not contacted.
e This gives a fast response.
Case 2: Object Not Found (Cache Miss)
e The cache acts as a client and sends an HTTP request to the origin server.
e The origin server sends the HTTP response back to the cache.
e The cache:
1. Stores a copy of the object
2. Forwards the object to the client
i This dual role explains why the cache acts as both a client and a server.

Where Web Caches Are Typically Installed
Web caches are commonly deployed by:
e ISPs (residential ISPs)
© Universities
e Companies and enterprises
They are placed close to users, reducing the distance data must travel.
Why Web Caching Is Used
a) Reduce Response Time
e Cached objects are served from a nearby cache, not a distant origin server.
e = This significantly lowers latency and improves user experience.
b) Reduce Network Traffic
e Fewer requests reach the institution's access link to the Internet.
e Saves bandwidth and reduces congestion.
c) Improve Internet Scalability
e The Internet is dense with caches, meaning:
e Popular content is replicated at many locations
e Origin servers are protected from overload
e Content providers can deliver content efficiently and at scale.
Key Takeaway
e A Web cache intercepts HTTP requests and serves cached content whenever possible.
e It improves performance, reduces traffic, and enhances scalability.
e This mechanism is fundamental to how the modern Internet efficiently delivers web content.

Trojan Horse
A Trojan horse is a malicious program that is hidden inside otherwise useful or legitimate software.
The user is tricked into installing or running it, believing it is safe. Once executed, the Trojan performs
harmful actions such as opening backdoors, stealing data, or installing other malware.
Trojan horses are generally non-self-replicating; they rely on user deception rather than spreading
themselves.
Virus
A virus is malware that attaches itself to a legitimate application or file.

Infection occurs when a user receives an infected object (such as an email attachment) and actively
executes it.
After execution, the virus runs its malicious code and self-replicates by attaching itself to other files or
programs, allowing it to spread to other hosts when those files are shared.
Worm
Aworm is a standalone, self-replicating malware program.

Unlike viruses, worms do not require user interaction. Infection happens when a host passively receives
data that automatically executes, often by exploiting network vulnerabilities.
Worms rapidly propagate across networks and typically aim to consume resources such as network
bandwidth, server buffers, and processing power, causing congestion and service disruption.

Key Idea

e Trojan horse: hides in useful software, no self-replication

e Virus: spreads via infected files, requires user execution

© Worm: spreads automatically over networks, no user action needed

Packet sniffing is a security issue that arises mainly in broadcast (shared) media networks, such as shared
Ethernet hubs or wireless networks (Wi-Fi).
What happens in packet sniffing?
e In broadcast media, every packet sent on the network is physically visible to all devices on that
medium.
e Normally, a network interface card (NIC) accepts only packets addressed to it.
e In promiscuous mode, the NIC is configured to read and record all packets, regardless of the
destination address.
Relation to the figure
e The packet shown has src: B, dest: A.
e Although it is intended only for host A, the packet travels over the shared medium.
e Amalicious host (the attacker laptop in red) can:
e Put its NIC in promiscuous mode
e Capture the packet, even though it is not the destination
e Read the payload, which may contain sensitive data
Why is this dangerous?
e Attackers can capture:
e Usernames and passwords
e Session cookies
e Emails and private data
e This is especially severe when data is sent in plaintext (e.g., HTTP, FTP, Telnet).
