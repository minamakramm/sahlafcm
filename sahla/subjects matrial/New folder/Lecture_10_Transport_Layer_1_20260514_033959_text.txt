@ Sender Side — Multiplexing
In the middle host (Host 2):
e Two applications (P1 and P2) want to send data at the same time.
e Each application uses its own socket (yellow blocks).
e The transport layer collects (multiplexes) data from both sockets.
e It adds a transport header (containing source and destination port numbers).
e The data segments are forwarded down the stack to the network, link, and physical layers for
transmission.
This process is called multiplexing because data from multiple sources are combined into a single
outgoing data stream.

[Image 2]  725×215px  (PNG)
OCR:
@ Receiver Side — Demultiplexing
At Host 1 and Host 3:

e Incoming segments arrive from the network to the transport layer.

e The transport layer examines the header information (mainly destination port numbers).

e Each segment is delivered to the correct socket.

e The socket forwards the data to the appropriate application process (P3 on Host 1 and P4 on Host 3).
This sorting and delivery is known as demultiplexing.

Feature Explanation

Connectionless No handshake or session setup between sender and receiver. Each UDP

@ How the UDP Checksum Works
At the Sender
1. The sender treats the entire UDP segment (header + data) as a sequence of 16-bit words.
2. It computes the 1’s complement sum of all these 16-bit values.
3. The 1's complement of the final sum becomes the checksum.
4. This checksum is inserted into the UDP checksum field before transmission.
At the Receiver
1. The receiver performs the same 1's complement summation over:
e The received segment
e Including the checksum field
2. It checks the result:
Result Meaning
Checksum matches No error detected — segment accepted
xX Mismatch Error detected — segment discarded

[Image 2]  469×290px  (PNG)
OCR:
Important Detail: Is the Checksum Perfect?
J Not 100% Reliable
Although the checksum detects most errors, it is not perfect:
e Rare error patterns can still occur that:
e Alter the data without changing the checksum result.
e This means:
e Some corrupted packets may pass undetected.
This possibility is extremely small but not zero.
This is why UDP says it "detects errors, but may miss some."

Sender and Receiver Finite State Machines
Even though the protocol is trivial, it is modeled using two finite state machines (FSMs)—one for the sender
and one for the receiver.
Sender FSM
e The sender waits for data from the application.
e When data arrives, it simply sends the packet into the channel.
e Since nothing goes wrong, the sender remains in the same state and continues sending new data as it
arrives.
Receiver FSM
e The receiver waits for a packet.
e When a packet arrives, it immediately delivers the data to the application.
e It then returns to its single state and waits for the next packet.
Because the channel never corrupts or loses packets, no feedback (ACK/NAK) is required.

How TCP Sets the Timeout Value (Based on RTT Behavior)
TCP must choose a retransmission timeout (RTO) that is longer than the Round-Trip Time (RTT).
However, RTT varies, so choosing this timeout is tricky:

If timeout is too short

e TCP assumes a packet is lost even though it is only delayed.

e This causes premature timeouts and unnecessary retransmissions.

e Wastes bandwidth and reduces performance.

If timeout is too long

e TCP waits too long before retransmitting a truly lost packet.

e This causes slow recovery from losses.

e Leads to poor throughput.

3. TCP uses an averaged (smoothed) RTT
Rather than relying on one measurement, TCP uses an exponential weighted moving average (EWMA):
EstimatedRIT = (1 — a) - EstimatedRIT + a- SampleRTT
This smooths out fluctuations.
4. Add a “safety margin” (RTT variation)
TCP also computes the RTT deviation, then sets:
Timeout = EstimatedRTT + 4-DevRTT

This ensures:

e In normal conditions: timeout is near RTT

e If RTT spikes: timeout increases automatically

1. Measure RTT using SampleRTT.

2. Smooth it using an average.

3. Measure variations in RTT.

4. Set TCP timeout = smoothed RIT + cushion for variation.

Summary: TCP Reliable Data Transfer
e TCP provides reliable delivery even though IP is unreliable.
e It achieves reliability using:
e Pipelined segments: multiple unacknowledged packets can be in flight at once.

TCP Sender: How It Works
1. When the application gives data to TCP
e TCP takes the data and creates a segment.
e It assigns a sequence number, which is the byte number of the first byte in that segment.
e If TCP is not already waiting for an ACK, it starts a timer.
e This timer is always for the oldest unacknowledged segment.
e Its duration is the Timeoutinterval (based on EstimatedRTT).
2. When the timer expires (timeout event)
e TCP assumes the segment was lost.
e It retransmits that same segment.
e Then it restarts the timer (again for the oldest unacked segment).
This ensures lost data is resent.
3. When an ACK arrives
e If the ACK confirms receipt of new data (not previously acknowledged):
¢ TCP updates the boundary of what has been successfully received.
e If there are still unacknowledged segments:
e TCP keeps the timer running, or restarts it if needed.
e If all data is acknowledged, the timer stops.
In short
e Send data — start timer.
e Timeout — retransmit and restart timer.
e ACK — update progress and manage the timer.
This process ensures TCP reliably delivers data over an unreliable network.

Lost ACK — Detailed Summary
1. Sender transmits a data packet with a sequence number (e.g., seq = 92).
2. Receiver successfully receives the packet, delivers the data, and prepares an ACK.
3. Receiver sends ACK(seq) to confirm correct reception.

7. Sender retransmits the same data packet.
8. Receiver receives a duplicate packet, recognizes the repeated sequence number.
9. Receiver discards the duplicate data to avoid delivering the same data twice.
10. Receiver sends another ACK(seq).
11. Sender receives this ACK and moves on to the next packet.
