A burst error is more likely to occur than a single-bit error because the duration of
the noise signal is normally longer than the duration of 1 bit, which means that when

affect 10,000 bits.

Question
Is it effective to implement error
detection and correction techniques in 
synchronous or asynchronous 
communication systems ?
It is *absolutely effective and necessary* to implement error 
detection in both systems. Implementing error correction is most 
effective and provides the greatest benefit in *synchronous 
communication systems* where minimizing latency and managing 
noise on high-speed links is crucial.

Single Bit Parity Example
Let’s say we are using even parity (total number of 1s must be 
even).
Example: Data bits: 1010001 Count of 1s = 3 (odd number)
👉To make it even, we add parity bit = 1
Final transmitted data: 10100011 
↑ last bit = parity bit
--- At the receiver: Received bits = 10100011 
Count of 1s = 4 (even) → ✅No error detected.
If one bit flips during transmission, e.g. 11100011, Count of 1s = 5 
(odd) → ❌Error detected.
🟡Note: Single parity detects an error but cannot locate or correct 
it.

1. Protocols for Noiseless Channels
A noiseless channel is an ideal communication channel where no data 
is lost or corrupted during transmission.
Since there are no errors, these protocols do not need error detection 
or retransmission mechanisms.
Types:
• Simplest Protocol:
• The sender transmits frames continuously without waiting for 
acknowledgment.
• The receiver automatically accepts every frame correctly.
• Used only in theoretical or error-free environments.
• Stop-and-Wait Protocol (Noiseless):
• The sender transmits one frame and then waits for an 
acknowledgment (ACK) before sending the next frame.
• Ensures orderly delivery and synchronization even in error-free 
conditions.

────────────────────────────────────────────────────────────
PAGE 27  |  140 words
────────────────────────────────────────────────────────────
2. Protocols for Noisy Channels
A noisy channel introduces errors, losses, or distortions in transmitted 
frames. Hence, these protocols include error detection and retransmission 
(ARQ – Automatic Repeat Request) mechanisms.
Types:
•Stop-and-Wait ARQ:
• Sender sends one frame and waits for an ACK.
• If the ACK is not received within a timeout period, the sender 
retransmits the frame.
• Simple but inefficient for long-distance or high-delay networks.
•Go-Back-N ARQ:
• The sender can send multiple frames (up to a window size N) before 
receiving ACKs.
• If an error is detected, all frames from the erroneous one onward are 
retransmitted.
• More efficient but may lead to redundant retransmissions.
•Selective Repeat ARQ:
• Only the erroneous frames are retransmitted, not the entire batch.
• Uses individual ACKs for each frame.
• Most efficient but requires more memory and processing.

•The sender receives data from the 
network layer and places it in a data frame 
with a sequence number (Sₙ).
•It sends this frame to the receiver through 
the physical layer.
•After sending, the sender waits for an 
acknowledgment (ACK) from the receiver.
•If the ACK arrives correctly and in time → 
the sender proceeds to the next frame 
(Sₙ+1).
•If no ACK is received within a certain time 
period → a timeout occurs, and the sender 
retransmits the same frame.

Flow of Operation (Step-by-Step 
Example)
1.Sender → Receiver: Sends frame 0.
2.Receiver → Sender: Sends ACK 0 
(meaning frame 0 received correctly, 
now expecting frame 1).
3.Sender: Upon receiving ACK 0, sends 
frame 1.
4.If ACK is lost: Sender waits for 
timeout, then retransmits frame 1.
5.Receiver: Detects it’s a duplicate, 
discards it, and resends ACK 1.
Purpose of Sequence Numbers
The sequence numbers (0 and 1) prevent confusion between:
•New frames
•Retransmitted frames (in case of acknowledgment loss)
This is why the Stop-and-Wait ARQ uses 1-bit sequence numbers (alternating 0 and 
1).

e Both sender and receiver run continuous loops (“Repeat forever”) listening for events:
© Request from network layer — send new data.
© Notification from physical layer — frame arrival.
e Timeout event — retransmit frames (if no ACK received).
e The timers ensure reliability by detecting lost or delayed frames.
5. Data Flow Summary
1. Sender transmits multiple frames (within window size).
2. Receiver acknowledges the last correctly received frame via piggybacking.
3. If an error occurs (e.g., frame 3 lost), receiver discards frames 4-5.
4. Sender times out and goes back to retransmit frame 3 onward.
@ 6. Advantages of Go-Back-N with Piggybacking
Feature Description
Efficiency Multiple frames sent without waiting for ACKs.
Bandwidth Saving ACKs combined with data (piggybacked).
Simplified Receiver Only accepts in-order frames.
Drawback If one frame is lost, all following frames in the window must be resent.
