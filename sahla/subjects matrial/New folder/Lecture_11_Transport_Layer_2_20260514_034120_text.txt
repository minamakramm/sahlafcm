TCP flow control ensures that a fast sender does not overwhelm a slow receiver. The key idea is to match
the sender's rate to the rate at which the receiving application can read data.
1. Receive buffer at the TCP receiver
e Each TCP receiver allocates a receive buffer (RcvBuffer) in memory.
e Incoming TCP data from IP is first stored in this buffer.
e The application process reads data from the buffer at its own speed (which may be slow).
2. Why flow control is needed
e If the application reads data slowly, the buffer can fill up.
e If the sender keeps transmitting data too fast, the receive buffer may overflow.
e TCP prevents this using flow control.
3. Buffer structure (as shown in the figure)
The receive buffer consists of:
e TCP data in buffer (green shaded area):

Data already received but not yet read by the application.
e Spare room (blue area):

Free space available for new incoming TCP data.
So:

RevBuffer = TCP data in buffer + Spare room
4. Receiver Window (RcvWindow)
e The Receiver Window (RcvWindow) is defined as:
RevWindow = sparc room in the receive buffer

e It tells the sender how much more data the receiver can accept without overflowing.

5. How the receiver advertises RcvWindow
e The receiver includes the value of RcvWindow in every TCP segment (ACK) it sends.
e This value dynamically changes as:
e The application reads data — buffer frees space — RcvWindow increases
e New data arrives — buffer fills - RcvWindow decreases
6. Sender behavior (flow control rule)
e The sender must ensure:
UnACKced data < RevWindow
© That is, the sender limits the amount of outstanding (unacknowledged) data to what the receiver can
handle.
7. Guarantee provided by TCP flow control
This mechanism guarantees:
e No receive buffer overflow
© Speed matching between:
e Sender's transmission rate
e Receiver application's data consumption (drain) rate
8. Important assumption (from your note)
e Ifthe TCP receiver discards out-of-order segments, flow control still works because:
e Only in-order, valid data occupies buffer space
e RcvWindow accurately reflects usable buffer capacity
9. One-line summary
TCP flow control protects the receiver by advertising available buffer space (RcvWindow) and forcing
the sender to transmit only as fast as the receiving application can consume data.

Scenario 3: Four senders, multihop shared paths, timeouts & retransmissions
What's happening
e There are multiple senders (four flows) transmitting data across a multihop network.
© Several flows share the same links and routers.
e Routers have finite output buffers (limited queue space).
e As input rate A,,, increases, queues build up and packets are dropped.
Role of timeouts and retransmissions (Host A)
e When congestion causes packet loss, the sender (e.g., Host A) does not know immediately why the
packet was lost.
e TCP assumes loss = congestion and retransmits the packet after a timeout.
e Now the network carries:
e Original packets
e Retransmitted packets
e This increases the offered load, making congestion even worse.
So congestion causes retransmissions, and retransmissions amplify congestion.

The “hidden cost” of congestion (key idea)
When a packet is dropped, all upstream transmission capacity used to carry that packet is wasted.

Why this matters

e A packet may traverse several links successfully before being dropped near the destination.

e All bandwidth used on earlier links for that packet is completely wasted.

e With multihop paths, this waste can be substantial.

Example:

e Packet travels 4 hops

e Dropped at hop 4

© Capacity of hops 1-3 was wasted

1. Throughput (Goodput)
Definition
e Throughput refers to the rate of successful data delivery from sender to receiver.
e More precisely, congestion control aims to maximize goodput, not just raw sending rate.
Goodput
© Goodput = useful data delivered end-to-end
e Excludes:
e Retransmitted packets
© Control packets
© Dropped packets
Why it matters
e Sending faster does not always mean higher performance.
e Under congestion, excessive sending causes losses and retransmissions, reducing useful throughput.
Goal
e Keep the network operating near its optimal point so that useful data delivery is maximized.

Fairness and UDP / Parallel TCP Connections -
e TCP fairness:
TCP congestion control is designed so that multiple TCP connections sharing a bottleneck link get a fair
share of bandwidth.
e@ Why multimedia avoids TCP:
Real-time audio/video applications do not want their sending rate reduced by TCP congestion control.
Instead, they use UDP, sending data at a constant rate and tolerating some packet loss, which is
preferable to delays and jitter.
e Fairness issue with UDP:
Since UDP has no congestion control, it can consume more bandwidth than TCP flows, leading to
unfairness.
e Parallel TCP connections:
Nothing prevents an application from opening multiple TCP connections between the same two hosts.
e Effect on fairness:
Each TCP connection gets its own fair share, so using parallel connections increases total bandwidth
share.
e Example:
Web browsers open multiple TCP connections to download different objects of a webpage faster,
effectively getting more bandwidth.
In short:
TCP enforces fairness per connection, but UDP flows and parallel TCP connections can bypass this fairness,
potentially dominating network resources.

Socket = Communication Door
e Asocket is the interface between a process (application) and the transport layer.
e The application developer controls the process behavior (what to send, when to send).
e The Operating System (OS) controls the transport infrastructure (TCP/UDP, routing, delivery).
Sending a Message (Door Analogy)
1. The sending process wants to communicate.
2. It writes (pushes) a message into its socket
— like shoving a message out of a door.
3. After that point, the process does not worry about how the message travels.
4. The transport layer (TCP/UDP) and lower layers (IP, link layer), managed by the OS, take responsibility.
Transport Infrastructure (Behind the Door)
e The sending process relies on the transport infrastructure:
e Breaks data into segments
e Adds headers (ports, sequence numbers, etc.)
e Moves data across the Internet
e All of this is controlled by the OS, not the application.
Receiving a Message
1. The transport infrastructure delivers the message to the destination host.
2. The OS places the message into the receiving process's socket.
3. The receiving process reads the message from its socket
— like opening the door and taking the message inside.

Why Processes Need an Identifier
e For a process to receive messages, it must have a unique identifier.
e The network must know exactly which process on which host should get the data.
IP Address: Identifies the Host
e Every host device on the Internet has a unique 32-bit IP address (IPv4).
e The IP address identifies the machine, not the specific process.
? Question:
Is the IP address alone sufficient to identify a process?
Answer: No.
e Asingle host can run many processes at the same time (web server, mail server, browser, etc.).
Port Number: Identifies the Process
e To identify a specific process on a host, we use a port number.
e A port number is a 16-bit number assigned to a process by the OS.
Process Identifier = IP Address + Port Number
e Apracess on the Internet is uniquely identified by:
(IP address, Port number)
e This combination ensures messages reach the correct application on the correct host.

Loss-tolerant means:
The application can still function correctly even if some data (packets/messages) are lost during
transmission.

e Some packets may never arrive.

e The application does not require retransmission of lost data.
